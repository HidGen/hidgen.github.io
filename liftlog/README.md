# Подходы — лёгкий офлайн-дневник качалки

PWA на ванили (HTML/CSS/JS) + IndexedDB. Без сборки, без бэкенда: вся история
тренировок хранится **на устройстве**, работает офлайн. Интерфейс — под усталые руки:
крупные кнопки, никакой клавиатуры, новый подход по умолчанию повторяет прошлый.

## Запуск
Нужен любой статический сервер (для Service Worker — контекст `localhost` или `https`).
```bash
cd liftlog
python -m http.server 8080
# открыть http://localhost:8080
```
На телефоне для полноценного PWA (установка на экран, офлайн-кэш) нужен **HTTPS**
(например, GitHub Pages). По чистому HTTP приложение тоже работает (IndexedDB доступен),
но без установки и офлайн-кэша оболочки.

## Флоу
1. **Начать тренировку**.
2. **+ Упражнение** → имя (чип из недавних или ввод) + стартовый вес.
3. На экране упражнения: степперы веса (±2.5) и повторов (±1), большая кнопка
   **«Записать N × R»**. Тот же подход — один тап; изменил — пара тапов по ±.
4. **Готово** → упражнение в списке тренировки. **Завершить тренировку** → в Историю.

## Контракт хранения (IndexedDB `liftlog`, расширяемый)
Упражнение имеет **тип `kind`**, а подход — открытую карту **`values`**, поэтому модель
не привязана к «вес/повторы». Типы и их поля описаны в реестре `KINDS` (в `app.js`) —
добавить новый тип (плавание, велотренажёр…) = дописать запись в реестр, без миграций.

- `workouts` {id, startedAt, finishedAt, …sync}
- `exercises` {id, workoutId, name, **kind**, startedAt, …sync}
- `entries` {id, exerciseId, idx, **values{}**, loggedAt, …sync}
- `catalog` {name, kind, lastValues, lastUsed} — память для чипов

Типы из коробки: `strength` (вес+повторы), `bodyweight` (повторы+доп.вес),
`cardio` (время, дистанция, ср.пульс, нагрузка), `timed` (время).

## Синхронизация через GitHub (без своего сервера)
«Бэкенд» — сам GitHub: приложение через `api.github.com` читает/пишет
`athletes/<id>.json` в приватный репозиторий данных. Сервер запускать не нужно.

Настройка (один раз, в ⚙ Настройках):
1. Создай приватный репо данных, напр. `HidGen/liftlog-data` (пустой).
2. Fine-grained PAT: Settings → Developer settings → Fine-grained tokens →
   только репо `liftlog-data` → Permissions **Contents: Read and write**.
3. В ⚙ вставь `owner/name` репо и токен → «⟳ Синхронизировать».

Слияние **last-write-wins** по `updatedAt`. Качок синкает свои данные; тренер по ID
тянет/ведёт данные подопечных (общий репо данных в кругу доверия тренер↔качки).

> ⚠️ Токен лежит в `localStorage` устройства (статика не умеет иначе). Берём
> fine-grained PAT только на один репо данных (Contents) — утечка ограничена им.
> Для прода — нормальная авторизация/бэкенд.

## Роли (раздельные стартовые экраны)
- **ID качка** генерится при первом запуске (`localStorage`), показан в ⚙ — качок даёт его тренеру.
- 🏋 **Качок** — стартовый экран: последние тренировки + «Начать тренировку».
- 📋 **Тренер** — стартовый экран: список подопечных. Тап по подопечному → его история
  и возможность **вести тренировку за него** (`owner` записи = ID подопечного), удобно
  заполнять с устройства тренера, если качку несподручно.
- Данные с владельцем (`owner`), мягкое удаление, слияние **LWW** — контракт под синк без миграций.
- Бэкап локально — **Экспорт/Импорт JSON** в ⚙.

## Локальный прогон
```bash
cd liftlog && python -m http.server 8080      # http://localhost:8080
```
Без токена — работает только на устройстве. С токеном GitHub (⚙) — синк качок↔тренер.

> На GitHub Pages приложение на HTTPS, а GitHub API — тоже HTTPS, поэтому синк
> работает и в проде (в отличие от HTTP-стенда — там был бы mixed content).

## Прежний серверный вариант
В `sync-server/` лежит альтернативный синк на своём сервере (Python+SQLite) — на
случай, если захочется уйти от GitHub-токена к нормальному бэкенду.
